# 修复 FTP 正常、SD 乱码的文件名显示问题 规格说明

## 背景

- 现象：同一批文件在 FTP 目录中显示文件名正常，但复制到 SD 卡目录后，部分文件名显示为乱码符号。
- 影响：用户无法稳定识别文件，目录浏览与检索体验下降，容易误判文件损坏。

## 目标

- 保证“文件名显示”在 FTP 与 SD 两种来源下具备一致性和可预期性。
- 避免把编码不稳定的标题文本误当作文件名展示。
- 保留现有功能（排序、搜索、打开、收藏、历史）不退化。

## 根因分析（现状）

- 本地/SD 列表显示逻辑会优先使用 `displayTitleByPath`，当标题缓存存在时会覆盖文件名显示。
- FTP 列表不走标题覆盖逻辑，因此同一文件在 FTP 看起来“文件名正常”，到 SD 后被标题替换为乱码，形成“来源差异”。
- MHTML 标题提取虽已有编码探测，但仍存在部分样本提取失败或提取到脏文本的情况，导致本地展示异常放大。
- FTP 编码链路仍存在“解码与 URL 重新编码不一致”的风险点，可能引入个别名称的二次编码偏差。

## 变更范围

### 包含

- 文件列表展示策略调整：主显示位固定展示真实文件名，不再被标题覆盖。
- 标题展示降级：标题作为附加信息展示，并增加乱码判定过滤。
- 标题缓存写入前质量门禁：检测明显乱码/不可见控制字符，命中则不覆盖展示。
- FTP 编码一致性修正：统一同一路径中“解码 charset”与“重编码 charset”的来源，降低二次编码偏差。

### 不包含

- 修改文件真实存储名（不做批量重命名迁移）。
- 放开网络资源策略或改造阅读器内核。
- 重写整个 FTP 实现（仅做兼容性增强）。

## 方案设计

### 1) 展示语义解耦：文件名与标题分离

- 列表第一行固定显示 `entry.name`（真实文件名语义）。
- 提取到的标题放入第二行附加展示（如 `Title: ...`），不参与文件名主展示位替换。
- 这样即使标题乱码，也不会被误认为“文件名乱码”。

### 2) 标题防乱码门禁

- 在标题提取结果入缓存或用于 UI 前，执行质量校验：
  - 替换符 `�` 占比阈值校验；
  - 不可见控制字符校验；
  - 极端异常符号密度校验。
- 命中异常则回退为空标题，仅显示文件名与元信息。

### 3) FTP 编码一致性

- 对 FTP “名称解码”和“路径构造编码”统一使用同一 charset 决策来源（显式配置优先，Auto 下使用会话探测结果）。
- 避免 Auto 模式下一端按 GBK 解码、另一端按 UTF-8 重编码导致路径/名称错位。

### 4) 可观测性与诊断

- 增加调试日志字段：
  - 当前来源（FTP/LOCAL）
  - 文件名展示值、标题展示值
  - 标题门禁是否命中
  - FTP charset 决策结果
- 日志仅用于调试，不输出敏感账号信息。

## 实施点位（预期）

- `app/src/main/java/html_reader/FilesFragment.kt`
  - 列表 `getView` 显示策略改造（文件名主显、标题附显）。
  - FTP 编码决策与路径构造一致性修正。
- `core/title/impl/HtmlTitleExtractor.kt`（或标题展示调用侧）
  - 增加标题质量门禁/异常过滤。
- `core/data/repo/TitleCacheRepository` 调用侧
  - 对异常标题结果进行缓存写入保护。

## 风险与兼容性

- 视觉层面会从“标题优先”调整为“文件名优先”，需确保用户可接受并在规格中明确。
- 标题门禁阈值过严可能丢失部分正常非拉丁文本，需要通过样本回归校准。
- FTP Auto 编码策略需避免对当前正常样本产生回归。

## 验收标准

- 同一文件在 FTP 与 SD 下的“文件名主显示”一致且可读，不出现来源切换后乱码突变。
- 标题乱码不会覆盖文件名显示。
- 正常标题样本仍可展示，且不影响排序、搜索、打开流程。
- FTP 中文/日文/韩文等多语言文件名在 Auto 与显式编码配置下均可稳定浏览。
