# 修复 MHTML 被强制窄屏渲染 规格说明

## 背景

- 当前问题：部分桌面网页保存的 MHTML 在 App 内打开后会呈现为窄屏移动版，图片仅显示左侧局部；另一些 MHTML 又能正常桌面渲染并支持缩放。
- 该问题影响阅读完整性与可用性，尤其在图片、表格、宽布局内容上明显。

## 目标

- 在不破坏现有安全策略（本地内容禁止外链加载）的前提下，显著降低 MHTML 被错误移动化渲染的概率。
- 让桌面网页保存的 MHTML 在默认情况下优先以“可缩放的桌面布局”呈现。
- 保持现有 Reader 功能（标签、进度、外链拦截、缩放手势）不退化。

## 根因分析（基于现状代码）

- WebView 当前未配置桌面渲染关键参数（桌面 UA、宽视口、总览缩放等），默认行为更偏移动端。
- MHTML 渲染时拦截外链资源是正确策略，但对依赖外链样式的页面会造成样式退化；退化后更容易表现为窄屏/移动化。
- 当前缺少“渲染模式判定与回退”机制，同一套默认配置无法兼容不同来源的 MHTML 内容特征。

## 变更范围

### 包含

- Reader WebView 配置策略增强（MHTML 专用渲染参数）。
- MHTML 主文档加载前后的渲染策略注入（非业务逻辑层变更）。
- 渲染失败或窄屏异常时的自动回退策略（一次重试，不死循环）。
- 必要的配置开关与日志点，便于后续定位兼容问题。

### 不包含

- 放开 MHTML 外链资源加载（仍保持拦截策略）。
- 重构为新阅读内核或替换 WebView。
- 修改文件缓存、标签管理、历史记录等无关能力。

## 方案设计

### 1) 引入 MHTML 渲染策略（Desktop-First）

- 为本地 MHTML/HTML 渲染引入独立配置档位：
  - `useWideViewPort = true`
  - `loadWithOverviewMode = true`
  - `textZoom = 100`
  - `setInitialScale(0)`（由 WebView 自动选择初始缩放）
  - `setSupportZoom(true)` 与 `builtInZoomControls(true)` 保持
- 默认对 MHTML 启用“桌面优先” UA（Desktop UA），避免站点按移动端规则布局。
- WEB 普通网页与 PDF 维持现有行为，不强行复用该策略。

### 2) 增加窄屏异常自动回退

- 在首帧渲染后采集关键指标：`contentWidth`、`scale`、`viewWidth`。
- 若命中窄屏判定（如 `contentWidth * scale` 显著小于可视宽度且可确认非空页），执行一次“回退重载”：
  - 切换到备用策略（如 Auto UA 或保留桌面 UA但调整 overview 行为）。
  - 仅重试一次，防止循环重载。

### 3) 保持安全边界

- 继续阻断非本地外链资源加载，不因渲染优化放开网络资源。
- 对 `cid:`、`file:`、`content:`、`data:` 维持既有白名单策略。

### 4) 可观测性

- 在调试日志中记录：
  - 当前渲染策略档位
  - 首帧关键宽度参数
  - 是否触发回退及回退后结果
- 日志仅用于开发定位，不输出敏感路径与账号信息。

## 实施点位（预期）

- `core/reader/web/WebViewConfigurator.kt`
  - 扩展配置函数，支持按文件类型/渲染策略应用不同 WebSettings。
- `app/src/main/java/html_reader/ReaderFragment.kt`
  - 在加载 MHTML 时应用策略、采集首帧指标、执行一次回退重试。
- `core/reader/web/BlockingResourceWebViewClient.kt`
  - 保持拦截边界不变，仅补充必要可观测信息。

## 兼容性与风险

- 部分页面可能依赖移动端 CSS 才可读，Desktop-First 可能带来个别页面字体偏小，需要保留缩放可用性。
- WebView 设备差异存在，回退判定阈值需保守，避免误判引发闪屏重载。
- 必须确保回退至多一次，避免性能抖动与用户感知卡顿。

## 验收标准

- 问题样例 MHTML 打开后默认不再被强制窄屏，图片可完整查看并可缩放。
- 原本正常渲染的 MHTML 不出现明显退化。
- 外链资源仍保持拦截，不引入安全回归。
- 首次打开与标签切换流程无崩溃、无无限重载。
